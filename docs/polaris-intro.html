背景 · 问题

一个被忽略的断层

现有工具都围绕同一个人设计：开发者。最懂业务的人——产品经理——没有工具能直接表达需求。

当前市场

  • Dify / Coze — 拖拽搭建，产出平台配置，需技术操作
  • LangGraph / CrewAI — 代码框架，需开发者编排
  • Claude Code / Cursor — 编码助手，需开发者指挥
  • Copilot Studio — 企业级，仍依赖技术配置

被忽略的角色

  • 产品经理 懂业务、懂需求，没有直接表达的工具
  • 业务需求→开发指令之间，缺一个 翻译层
  • 工程师花大量时间理解需求，而非写代码
  • 需求文档和最终产出之间缺乏可追溯链路
解决方案

星云如何填补这个断层

核心竞争力

四个不可替代的差异

🎯

产出可运行的独立代码

Dify/Coze 产出平台配置——离开平台无法运行。星云产出完整代码库，可 Git 管理、可部署。这是"工具"和"工厂"的本质区别。

🏃

代码沙箱：生成即运行

代码生成不是终点。星云提供在线沙箱，生成后即可在浏览器运行。用户直接修改源码实时生效，告别"改配置→重发布"循环。

📄

文档驱动的编码质量

不是简单甩需求给 AI。先沉淀结构化文档（proposal→design→acceptance），让 AI 理解业务上下文后再编码。需求变更可追溯。

👤

站在 PM 的视角

所有平台以开发者为中心。星云使用者是讲业务需求的产品经理。平台承担"需求→指令"的翻译。这是产品出发点的根本差异。

部署模型

工厂，不是运行环境

星云是工厂，不是运行环境。客户买的是软件资产，不是平台订阅。

🏗 nebula-platform · Build Mode

星云平台完整版，构建+运行时引擎全量部署。PM 完成需求对话、文档生成、编码调度、构建验证。

PM 对话 Agent 文档生成 Skill 匹配 编码调度 构建验证 沙箱运行

🚀 nebula-runtime · Run Mode

两个独立代码库：星云给客户的不是"一个平台减配"，而是一个干净的运行平台。客户部署时只有 nebula-runtime + Build Artifact + 必要服务。

LangGraph 集群 MCP / A2A Gateway Build Artifact Postgres / Redis

不是"同一个平台的不同模式"。nebula-platform 和 nebula-runtime 是两个独立的部署单元。客户拿到的 nebula-runtime 没有任何构建引擎和平台管理逻辑，绝对干净。运行平台可公开镜像，客户也能审计。

Python 构建 = 验证 + 打包。星云后端是 Python，产出也是 Python——不需要编译。构建容器在编码完成后验证依赖安装、模块导入、跑测试，然后打出版本化的 Build Artifact。Artifact 可被 nebula-runtime 直接动态加载运行。
