# Brainstorming Session — 系统日志能力

- **日期**: 2026-07-12
- **主题**: 为 Nebula 系统增加日志能力
- **Change**: `add-system-logging`

---

## 讨论过程

### 1. 核心意图
为系统增加日志能力，以便开发过程中分析系统行为、定位问题。

### 2. 范围选择
- **后端 (Python/FastAPI)**: 需要重新配置日志系统 — 现有 5 处 `logger.*()` 调用因缺少日志配置而静默丢弃
- **前端 (React)**: 增加错误捕获和 API 层日志记录
- **全栈推进**: 优先后端日志配置和中间件，同步推进前端基础错误处理

### 3. 需求细化（用户确认）

| 问题 | 用户回答 |
|------|---------|
| 后端/前端都加还是选一个？ | **都加** |
| 主要用途是什么？ | **调试/排错** — 开发时追踪问题 |
| 输出目标？ | **文件** |
| 不做哪些？（分布式追踪/告警/行为分析） | 第一版都不做 |

### 4. 技术现状（代码探索发现）

**后端现状**:
- 使用标准库 `logging.getLogger(__name__)`，但**无任何日志配置**
- 全局异常处理程序捕获异常但不记录
- 无请求/响应日志中间件
- 无日志级别、格式配置
- `.env` 无 `LOG_LEVEL` 等变量

**前端现状**:
- 零 `console.log/error/warn` 调用
- 存在空 `catch {}` 块静默吞异常
- 无错误边界（Error Boundary）
- 无日志工具函数

---

## 关键决策

| # | 决策 | 说明 |
|---|------|------|
| 1 | 后端用标准库 `logging` + `TimedRotatingFileHandler` | 避免引入新依赖，标准库足够满足调试需求 |
| 2 | 后端日志格式为结构化文本（非 JSON 第一版） | 调试场景下人类可读 > 机器解析 |
| 3 | 前端日志通过 API 发送到后端写入文件 | 浏览器无法直接写文件，采用后端统一存储 |
| 4 | 前端增加 Error Boundary | 捕获 React 组件渲染错误并上报 |
| 5 | 日志级别通过环境变量 `LOG_LEVEL` 配置 | 默认 `INFO`，开发可调为 `DEBUG` |

---

## 需求要点（整理后）

### 后端日志系统
1. **日志配置引导**: 在 `app/main.py` 启动时初始化日志配置
2. **日志级别**: 通过环境变量 `LOG_LEVEL` 控制（`DEBUG`/`INFO`/`WARNING`/`ERROR`）
3. **输出目标**: 控制台（开发）+ 文件（按天轮转，保留 30 天）
4. **日志格式**: `时间 | 级别 | 模块 | 消息` + 异常 traceback
5. **请求日志中间件**: 记录每个请求的 method、path、status_code、duration
6. **异常日志**: 全局异常处理程序记录完整 traceback
7. **激活现有日志**: 确保现有 5 处 `logger.*()` 调用正确输出

### 前端日志系统
1. **日志工具函数**: `logger.info()/warn()/error()` 封装，统一输出格式
2. **API 客户端日志**: 记录请求 URL、方法、状态码、响应时间
3. **Error Boundary**: 捕获组件渲染错误并记录
4. **日志上报**: 前端日志通过 `/api/v1/logs` 发送到后端写入文件

### 配置文件变更
1. `.env` 增加 `LOG_LEVEL=INFO`, `LOG_DIR=./logs`
2. `.env.example` 同步更新

## 边界范围（明确不做）

1. ❌ 分布式追踪（OpenTelemetry / Jaeger）
2. ❌ 日志告警规则（如 Grafana Alerting）
3. ❌ 前端用户行为分析/事件埋点
4. ❌ 结构化 JSON 日志（第一版，后续可按需升级）
5. ❌ 外部日志服务集成（ELK / Loki / DataDog）
6. ❌ 日志查询界面
